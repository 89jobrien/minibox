02 Container Lifecycle State Machine
State transitions for a single container record in DaemonState. Driven by daemon events: spawn result, process exit, explicit stop/remove. Exit detection varies by platform — Linux uses waitpid; macOS/Windows use adapter futures or platform APIs.

04 Platform Adapter Selection Flow
At startup miniboxd detects the host platform and delegates to the appropriate platform crate. preflight() then selects a backend via MINIBOX_ADAPTER or capability probing. A fatal error is reported before the socket is bound if no backend is available.

08 Search Stack — searchbox + zoektbox
zoektbox owns the Zoekt binary lifecycle: download pinned release, verify SHA256, deploy to VPS via SSH. searchbox owns the SearchProvider port and adapters (Zoekt HTTP, fan-out, git mirror, filesystem rsync, local sidecar). searchboxd is a standalone MCP stdio server binary. The two crates have a strict one-way dependency: searchbox → zoektbox; zoektbox has no knowledge of search semantics.

09 Crux Pipeline Integration
.cruxx pipelines run inside minibox containers with structured tracing. TraceStore (port) lives in minibox-core; FileTraceStore adapter in minibox-agent. PipelineRunner orchestrates execution. minibox-crux-plugin is a standalone binary exposing container ops to crux via JSON-RPC. handle_pipeline in daemonbox wires it all together. A daemon-side recursion/depth gate prevents unbounded nesting.
