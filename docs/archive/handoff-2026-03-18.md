## Key decisions made

**winboxd is a Named Pipe proxy, not a daemonbox consumer.** Windows doesn't have
`tokio::net::UnixStream`, so `winboxd` relays bytes from `\\.\pipe\miniboxd` to `miniboxd`
inside WSL2 via `wsl.exe -- socat`. No `daemonbox` dependency on Windows. This avoids a large
conditional-compilation surface and keeps `daemonbox` Unix-only.

**`miniboxd/src/lib.rs` shim instead of test-file surgery.** Re-exporting `daemonbox::*` from
`miniboxd` let all existing tests and the `main.rs` continue to compile without any changes
during the migration, and serves as the permanent backward-compatibility surface.

**Per-crate `-p` flags everywhere.** `--workspace` commands always fail on one platform because
of the `compile_error!` guards in `miniboxd` (Linux only) and the future `macboxd` (macOS only).
House rule: never use `--workspace` for clippy, test, or check.

### Medium priority

3. **`macboxd` binary (Task 4 of macbox-daemonbox plan)** — wire Colima adapters + `daemonbox`
   into a macOS daemon binary. Needs macOS + Colima.

4. **Executor injection seam for WSL + Docker Desktop adapters** — `WslRuntime`,
   `WslFilesystem`, `WslLimiter` currently call `wsl.exe` directly with no injection point.
   Add `Arc<dyn Fn(&[&str]) -> Result<String>>` (same pattern as Colima's `LimaExecutor`) so
   they can be unit-tested without a real WSL installation.

### Lower priority / deferred

5. Compile-checked tracing helpers (macros/wrappers that enforce canonical field names at
   compile time) — the tracing contract is documented in CLAUDE.md; enforcement is still manual.

6. `MacboxdFixture` for macboxd e2e tests — build after macboxd binary exists.

7. Native Windows backend — `winboxd` WSL2 proxy is explicitly a stepping stone. A future
   `winbox-native` could use Hyper-V direct or WSL2 kernel APIs. No plan yet.

```bash

# Linux full unit suite
just test-unit

# Linux integration + e2e (root required)
```
